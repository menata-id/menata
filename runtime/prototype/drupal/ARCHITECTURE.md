# Drupal Prototype Architecture

> This document describes how Menata Runtime Metadata concepts map to native Drupal configuration.

---

## Overview

Drupal is a configuration-driven CMS framework built on PHP.

Its built-in modules provide most of what Menata Runtime requires — without any custom code:

- Entity model (Content Types + Fields API) → Machine + Field
- Workflow module → state machine + transitions
- ECA module (Events, Conditions, Actions) → event responses
- Views module → list and detail views
- Form API → form views (auto-generated)
- Roles + Permissions → permission enforcement
- CMI (Configuration Management Interface) → all of the above expressed as YAML

---

## Architecture

```text
Runtime Metadata (YAML)
        |
        v
Drupal native config YAML (generated by Menata Runtime interpreter)
        |
        |--> Content Type API   ---> Machine realization
        |--> Fields API         ---> Field realization
        |--> Workflow module    ---> Status transition enforcement
        |--> ECA module         ---> Event responses (notify, create record)
        |--> Views module       ---> List + Detail view realization
        |--> Form API           ---> Form view realization (auto-generated)
        `--> Roles + Permissions --> Permission enforcement
        |
        v
Running Drupal Application
```

The Menata Runtime interpreter (separate layer) generates the Drupal YAML config files.
Drupal's own engine then interprets them — no custom code inside Drupal required.

---

## Machine -> Content Type

Each Machine becomes a Drupal Content Type.

```
machine.id: mch_design_request  ->  Content Type: design_request
machine.name: Design Request     ->  Label: Design Request
```

---

## Field -> Drupal Field

Each Field becomes a Drupal field on the content type.

| Runtime Metadata Type | Drupal Field Type |
|----------------------|-------------------|
| text | Text (plain) |
| rich_text | Text (formatted, long) |
| number | Number (decimal) |
| money | Number (decimal) — formatted as currency |
| boolean | Boolean |
| date | Date |
| date_time | Datetime |
| user | Entity Reference -> User |
| file | File |
| value_list | List (text) |
| reference | Entity Reference -> Content Type |

---

## Event -> Workflow + ECA

Status-changing events use the **Workflow module** for transitions.
Notification actions use the **ECA module** (Events, Conditions, Actions).

```
evt_submit:
  actions:
    - set_field: { field: fld_status, value: Submitted }   -> Workflow transition: draft -> submitted
    - notify: { role: Designer }                           -> ECA rule: on transition -> notify Designer role
```

---

## Constraint -> Field Config + ECA Condition

Simple required constraints:
```
cst_title_required -> field config: required = TRUE on fld_title
```

Conditional constraints (not yet expressible in YAML — see Metadata Coverage in README):
```
cst_attachment_required_for_banner
  condition: fld_design_type = Banner 2:1
  -> requires PHP constraint plugin (outside metadata proof scope)
```

---

## Permission -> Drupal Role

```
permissions:
  - role: Requester  events: [evt_submit]
  - role: Designer   events: [evt_accept, evt_reject, evt_start, evt_complete]
```

Realized as Drupal Roles with workflow transition permissions per role.

---

## View -> Views + Display Modes

| Runtime Metadata Type | Drupal Realization |
|----------------------|--------------------|
| list | Drupal View (Views module YAML) |
| detail | Content display mode (auto-generated) |
| form | Node edit form (auto-generated from field config) |

---

## Why Drupal Validates the Menata Architecture

The Menata Runtime specification listed Drupal as an architectural benchmark for:

- metadata-driven configuration
- reusable plugin components
- entity model
- views abstraction
- configuration management

This proof demonstrates that Runtime Metadata maps naturally onto native Drupal configuration.

Drupal does not need to be modified. No custom modules required inside Drupal.
